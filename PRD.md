# logsee PRD

## 1. 제품 개요

`logsee`는 터미널에서 로그를 확인하기 위한 TUI 도구다. 사용자는 파이프나 리다이렉션으로 들어오는 표준 입력 로그와, 이미 존재하는 로그 파일을 같은 UI에서 탐색하고 필터링할 수 있어야 한다.

이 문서는 구현 요구사항의 기준 문서다. 기능 동작, 키 바인딩, 입력 모델, 설정 형식이 코드나 도움말과 충돌하면 이 문서를 우선한다.

## 2. 목표

- `stdio`와 파일 입력을 모두 지원한다.
- `stdio` 입력은 먼저 파일에 저장하고, UI는 그 파일을 Source of Truth(SOT)로 사용한다.
- 파일 입력은 원본 파일 자체를 SOT로 사용한다.
- 입력 로그 타입을 자동으로 판별한다.
- 기본 지원 로그 타입은 `plain`, `adb`, `kernel`이다.
- 로그 타입 판별 패턴은 `config.toml`에서 설정할 수 있다.
- 필터 입력 진입 키는 `:`만 사용한다.
- 대용량 로그에서도 화면에 필요한 범위만 읽어 TUI 반응성을 유지한다.

## 3. 비목표

- 원격 로그 백엔드, 웹 UI, 데이터베이스 저장소는 v1 범위가 아니다.
- 바이너리 로그 포맷 직접 파싱은 v1 범위가 아니다.
- `stdio` 입력의 원문 저장 파일을 별도 포맷으로 변환하지 않는다. 저장 파일은 원문 텍스트 로그다.
- 필터 입력 진입 키로 `Enter`를 사용하지 않는다.

## 4. 입력 모델

### 4.1 입력 종류

`logsee`는 두 가지 입력 경로를 지원한다.

| 입력 | 예시 | SOT |
|------|------|-----|
| `stdio` | `some-command 2>&1 \| logsee` | `logsee`가 생성한 append-only 로그 파일 |
| 파일 | `logsee /path/to/log.txt` | 사용자가 지정한 원본 파일 |

인자 없이 실행하거나 입력 경로가 `-`이면 `stdio` 입력으로 처리한다. 경로 인자가 있으면 파일 입력으로 처리한다.

### 4.2 `stdio` 입력 SOT 모델

`stdio` 입력에서는 표준 입력에서 받은 로그를 UI가 직접 SOT로 삼지 않는다. 별도 writer worker가 먼저 파일에 원문 줄을 append하고, UI는 그 파일을 읽는 file-backed source를 통해 화면을 갱신한다.

필수 규칙:

- writer worker는 수신한 원문 줄을 순서대로 SOT 파일에 append한다.
- append된 줄은 flush/sync가 끝난 뒤 UI에서 읽을 수 있는 대상으로 간주한다.
- UI의 탐색, 필터링, 로그 타입 판별, tail follow는 모두 SOT 파일 기준으로 동작한다.
- 메모리 버퍼는 화면 반응성을 위한 cache일 뿐이며, SOT가 아니다.
- 같은 로그 줄은 UI에 표시되기 전에 SOT 파일에 기록되어야 한다.

기본 저장 파일명은 현재 작업 디렉터리 기준 `logsee-YYYYMMDD-HHMMSS.log`다. 사용자가 출력 경로 옵션을 제공하는 경우 해당 파일을 SOT로 사용한다.

### 4.3 파일 입력 모델

파일 입력에서는 지정된 파일 자체가 SOT다.

필수 규칙:

- 원본 파일을 별도 파일로 복사하거나 재저장하지 않는다.
- UI는 원본 파일을 window 단위로 읽어 표시한다.
- 대용량 파일을 전체 메모리에 올리지 않는다.
- 파일 내 줄 번호 또는 내부 sequence는 파일 기준 1-based 좌표로 다룬다.

## 5. 로그 타입 자동 판별

### 5.1 지원 타입

v1에서 지원하는 로그 타입은 다음과 같다.

| 타입 | 설명 |
|------|------|
| `plain` | 특별한 구조를 가정하지 않는 일반 텍스트 로그 |
| `adb` | Android logcat 계열 로그 |
| `kernel` | Linux kernel/syslog 계열 로그 |

### 5.2 판별 방식

기본 로그 타입은 `auto`다. `auto`에서는 SOT 파일의 초기 비어 있지 않은 N개 라인을 샘플링하고, 설정된 타입별 패턴과 비교해 점수를 계산한다.

판별 규칙:

- 각 로그 타입은 하나 이상의 정규식 패턴을 가질 수 있다.
- 샘플 라인이 특정 타입의 패턴과 매칭되면 해당 타입 점수를 올린다.
- 가장 높은 점수의 타입을 선택한다.
- 동률이거나 어떤 패턴도 매칭되지 않으면 `plain`을 선택한다.
- 한 번 확정된 로그 타입은 세션 중 임의로 변경하지 않는다.

### 5.3 기본 패턴 예시

구현은 내장 기본 패턴을 제공해야 한다. `config.toml`에 같은 타입의 패턴이 설정되면 설정값을 우선한다.

예시:

```toml
[log_type]
default = "auto"
probe_lines = 200

[log_type.patterns]
plain = []
adb = [
  '''^\d{2}-\d{2}\s+\d{2}:\d{2}:\d{2}\.\d+\s+\d+\s+\d+\s+[VDIWEF]\s+''',
  '''^[VDIWEF]/[^:]+:\s+'''
]
kernel = [
  '''^\[[\s\d.]+\]\s+''',
  '''^\w{3}\s+\d+\s+\d{2}:\d{2}:\d{2}\s+\S+\s+kernel:'''
]
```

## 6. 설정 파일

기본 설정 파일은 `$HOME/.local/logsee/config.toml`이다. 사용자가 `--config` 옵션을 제공하면 해당 경로를 우선한다.

설정 우선순위:

1. CLI 옵션
2. `config.toml`
3. 내장 기본값

필수 설정 항목:

| 항목 | 타입 | 기본값 | 설명 |
|------|------|--------|------|
| `log_type.default` | string | `"auto"` | 기본 로그 타입. `auto`, `plain`, `adb`, `kernel` 중 하나 |
| `log_type.probe_lines` | int | `200` | 자동 판별에 사용할 초기 비어 있지 않은 줄 수 |
| `log_type.patterns.<type>` | string array | 내장값 | 타입별 판별 정규식 목록 |

잘못된 설정은 명확한 오류 메시지로 보고해야 한다. 로그 타입 패턴이 잘못된 정규식이면 해당 설정 로드를 실패로 처리한다.

## 7. TUI 요구사항

### 7.1 기본 화면

TUI는 SOT에서 읽은 로그 줄을 목록으로 표시한다.

필수 정보:

- 현재 입력 모드: `stdio` 또는 `file`
- 현재 로그 타입: `plain`, `adb`, `kernel`
- 필터 입력창의 현재 필터 문자열
- 검색 입력창의 현재 검색 문자열
- 현재 줄 위치와 전체 줄 수
- follow 상태

#### 7.1.1 화면 구역

TUI 화면은 위에서 아래 순서로 다음 네 구역으로 구성한다.

| 구역 | 줄 수 | 내용 |
|------|------|------|
| 필터 입력창 | 1 | 적용 필터 문자열, 필터 편집 중인 입력값, 필터 파싱 오류 |
| 검색 입력창 | 1 | 현재 검색 문자열, 검색 편집 중인 입력값, 검색 입력 힌트 |
| 목록 화면 | `height - 3` | 필터 입력창, 검색 입력창, 상태바를 제외한 나머지 전체 영역. 로그 목록, 커서, 북마크 표시를 렌더링한다. |
| 상태바 | 1 | 입력, 줄 수, follow, wrap, out, 로그 타입 등 상태 정보 |

필수 규칙:

- 렌더 순서는 `필터 입력창` -> `검색 입력창` -> `목록 화면` -> `상태바`다.
- 목록 화면 높이는 터미널 전체 높이에서 필터 입력창 1줄, 검색 입력창 1줄, 상태바 1줄을 뺀 값이다.
- 표시할 로그가 없어도 목록 화면은 `height - 3` 줄을 빈 viewport로 유지하고 상태바는 항상 터미널 하단 줄에 표시한다.
- 필터 입력창과 검색 입력창의 label은 각각 `FILTER INPUT`, `SEARCH INPUT`처럼 입력 영역을 명확히 식별할 수 있는 문구와 충분한 대비로 표시한다.
- 필터 입력창, 검색 입력창, 상태바는 로그 본문과 구분되도록 전체 행 너비에 배경색을 적용한다.
- 편집 중인 입력창은 입력 문자열 끝에 커서 표시를 렌더링한다. 입력 문자열이 비어 있으면 빈 입력 위치에 커서 표시를 렌더링한다.
- 필터 파싱 오류는 필터 입력창에 표시하고 상태바에 중복 표시하지 않는다.
- 도움말 다이얼로그는 위 네 구역 위에 겹쳐 표시되는 modal overlay다.
- Bubble Tea 런타임은 full-screen alternate screen을 사용해 프레임의 첫 줄인 필터 입력창이 터미널 스크롤 위치와 무관하게 보이도록 한다.
- Bubble Tea 런타임의 `View()` 출력은 전체 화면 높이를 채우는 경우에도 마지막 줄 뒤 개행으로 화면을 스크롤하지 않아야 한다.

#### 7.1.2 로그 목록 화면

로그 목록 화면은 필터 입력창, 검색 입력창, 상태바를 제외한 나머지 전체 영역을 사용한다. 각 로그 행은 다음 순서로 출력한다.

```text
<로그라인넘버> <bookmark 번호> <로그>
```

필수 규칙:

- 로그라인넘버는 SOT 기준 1-based 줄 번호다.
- bookmark 번호 영역은 해당 로그 라인에 북마크가 있으면 `1`-`9` 번호를 표시한다.
- 북마크가 없는 라인은 bookmark 번호 영역을 공백으로 유지해 로그 본문 시작 위치가 흔들리지 않게 한다.
- 로그 본문은 원문 로그 텍스트를 표시한다.
- 커서가 위치한 로그 행은 다른 로그 행과 구분되게 강조한다.
- wrap 모드가 꺼져 있으면 긴 로그 본문은 현재 화면 폭 안에서 한 visual row로 제한한다.
- wrap 모드가 꺼져 있으면 `h`/`l` 키로 긴 로그 본문의 가시 구간을 좌우로 스크롤할 수 있다.
- wrap 모드가 켜져 있으면 긴 로그 본문은 현재 화면 폭에 맞춰 여러 visual row로 나누어 표시한다.
- wrap continuation row는 로그 본문 시작 열에 맞춰 들여쓰기하고, 로그라인넘버와 bookmark 번호를 반복하지 않는다.
- 좌우 스크롤은 로그 본문에만 적용하며 로그라인넘버, bookmark 번호, 선택 표시, 커서 위치는 변경하지 않는다.
- 좌우 스크롤 offset은 화면에 표시되는 모든 로그 본문에 동일하게 적용하며, 개별 로그 길이에 따라 줄별로 offset을 보정하지 않는다.

북마크 표시:

- bookmark 번호는 로그라인넘버와 로그 본문 사이의 고정 폭 영역에 표시한다.
- 북마크가 있는 줄은 번호 `1`-`9`를 한 자리 배지처럼 표시해 로그라인넘버와 시각적으로 분리한다.
- 북마크 배지는 전경/배경색을 본문과 다르게 사용한다.
- 커서가 위치한 행의 reverse/강조 스타일과 겹쳐도 북마크 번호를 식별할 수 있어야 한다.
- 북마크가 없는 줄도 동일 폭의 공백 영역을 확보해 로그 본문의 시작 열이 고정되어야 한다.

검색어 하이라이트 표시:

- 현재 검색 문자열이 비어 있으면 하이라이트를 표시하지 않는다.
- 현재 검색 문자열은 공백/탭 기준 다중 토큰으로 해석한다.
- 큰따옴표(`"..."`)로 감싼 검색어는 공백을 포함하는 하나의 토큰으로 처리한다.
- 검색어 토큰화에 실패하면 현재 검색 문자열 전체를 하나의 검색 토큰으로 사용한다.
- 한 로그 라인은 검색 토큰 중 하나라도 부분 일치하면 검색 매칭 라인이다.
- 화면 강조는 각 검색 토큰의 모든 부분 일치 구간에 적용한다.
- 같은 줄에서 하이라이트 구간이 겹치면 하나의 연속 구간처럼 병합해 표시한다.
- 검색 하이라이트와 `n`/`p` 매칭 이동은 같은 매칭 기준을 사용한다.
- 검색 하이라이트는 항상 대소문자를 구분한다.
- `--ignore-case`는 검색 하이라이트에 적용하지 않는다.
- 하이라이트는 로그 본문의 가시 영역에만 적용한다.

복사용 선택 표시:

- 복사용 선택은 연속 구간 선택과 Space pick 선택 두 종류를 지원한다.
- 연속 구간 선택은 선택 시작 앵커와 현재 커서 사이의 모든 로그 라인을 선택한 상태다.
- Space pick 선택은 `Space`로 개별 로그 라인을 토글한 비연속 선택 집합이다.
- 연속 구간 선택과 Space pick 선택은 동시에 존재할 수 있다.
- 연속 구간 선택 라인과 Space pick 선택 라인은 서로 다른 배경 톤이나 스타일로 구분한다.
- 한 줄이 연속 구간 선택과 Space pick 선택에 동시에 포함되면 연속 구간 선택 스타일을 우선한다.
- 커서가 위치한 선택 라인은 커서 강조와 선택 강조가 모두 식별 가능해야 한다.
- 선택 표시 단위는 화면에 줄바꿈되어 보이는 시각 행이 아니라 논리 로그 라인이다.

#### 7.1.3 상태바

상태바는 화면 하단 1줄에 고정하며, 입력 포커스를 받지 않는다. 상태바는 편집 중인 긴 필터/검색 문자열이 아니라 현재 세션 상태, 선택 상태, 일시 피드백만 표시한다.

필수 필드:

| 필드 | 예시 | 설명 |
|------|------|------|
| `lines` | `lines:120/9834` | 현재 커서가 가리키는 SOT 기준 줄 번호와 SOT 전체 줄 수 |
| `in` | `in:stdio:read` | 입력 모드와 읽기 상태. `stdio` 또는 `file`, `read` 또는 `eof` |
| `type` | `type:adb`, `type:auto~` | 확정된 로그 타입. 자동 판별 전이면 `auto~` |
| `follow` | `follow:on` | follow 모드 상태 |
| `wrap` | `wrap:off` | 로그 목록 wrap 모드 상태 |
| `out` | `out:session.log` 또는 `out:-` | `stdio` 입력 저장 파일. 파일 입력이면 `out:-` |
| `bm` | `bm:1,3,4` | 설정된 북마크 번호 목록. 없으면 `bm:-` |
| `sel` | `sel:3` | 연속 구간 선택과 Space pick 선택을 합친 선택 로그 줄 수 |
| `msg` | `msg:3 lines copied` | 복사 결과 등 짧게 표시되는 일시 메시지 |

상태바 표시 규칙:

- 기본 표시 순서는 `lines`, `in`, `type`, `follow`, `wrap`, `out`, `bm`, `sel`, `msg`다.
- 필드는 ` | ` 또는 동등한 짧은 구분자로 나눈다.
- 상태바는 항상 한 줄이며 줄바꿈하지 않는다.
- `out` 경로는 고정 최대 표시폭을 넘으면 터미널 폭과 무관하게 가운데 생략 처리한다.
- 터미널 폭이 부족하면 `out` 경로를 추가로 가운데 생략 처리한다.
- 그래도 폭이 부족하면 `bm`, `sel`, `msg` 순서로 오른쪽의 보조 필드를 생략할 수 있다.
- `lines`, `in`, `type`, `follow`, `wrap`은 가능한 한 마지막까지 유지한다.
- 상태바는 전체 행 너비에 배경색을 적용해 로그 목록 영역과 시각적으로 분리한다.
- `stdio` 입력의 전체 줄 수는 SOT 파일에 성공적으로 기록된 줄 수 기준으로 단조 증가한다.
- 파일 입력의 전체 줄 수는 원본 파일에서 확인 가능한 전체 줄 수 기준이다.
- `sel`은 선택이 없으면 생략한다.
- `bm`은 설정된 북마크 번호를 오름차순으로 표시하고, 북마크가 없으면 `bm:-`로 표시한다.
- `msg`는 복사 성공/실패, 클립보드 오류, `no lines`, `no match` 같은 토스트성 피드백에만 사용한다.
- 복사 결과 `msg`는 짧은 시간 동안 표시한 뒤 자동으로 지운다.
- 런타임 오류 같은 비복사 `msg`는 다음 키 입력 또는 다음 상태 변경 시 지울 수 있다.
- 복사 성공 메시지는 `N lines copied` 또는 `1 line copied`로 표시한다.
- 복사 대상이 없으면 `no lines`를 표시한다.
- 필터 파싱 오류는 필터 입력창에만 표시하고 상태바에 중복 표시하지 않는다.

상태바 시각 규칙:

- `follow:on`, `wrap:on`처럼 켜진 이진 상태의 값은 강조색으로 표시한다.
- `follow:off`, `wrap:off`처럼 꺼진 값은 회색 또는 낮은 대비로 표시한다.
- 라벨(`follow:`, `wrap:` 등)은 상태바 기본 전경색을 따른다.
- `msg`는 나머지 상태 필드와 구분되도록 별도 전경/배경색을 사용해 항상 강조한다.
- 복사 성공, 클립보드 오류, `no lines`, `no match` 등 모든 `msg` 값은 동일한 메시지 강조 스타일로 표시한다.

상태바 동작 규칙:

- 상태바는 렌더마다 현재 모델 상태에서 다시 계산한다.
- 상태바는 키 입력을 직접 소비하지 않는다.
- 도움말 다이얼로그가 열려도 상태바의 underlying 상태는 변경하지 않는다.
- 필터 입력창, 검색 입력창, 로그 목록 화면 중 어느 포커스에 있더라도 상태바는 같은 위치와 형식으로 유지된다.

### 7.2 탐색

v1 TUI는 최소한 다음 탐색을 지원한다.

| 키 | 동작 |
|----|------|
| `j` / `k` | 커서를 한 줄 아래/위로 이동 |
| `↑` / `↓` | `k` / `j`와 동일 |
| `Ctrl+F` / `Ctrl+B` | 커서를 한 페이지 아래/위로 이동 |
| `PageUp` / `PageDown` | `Ctrl+B` / `Ctrl+F`와 동일 |
| `Home` / `End` | 처음/끝으로 이동 |
| `G` | 가장 마지막 출력 로그로 이동하고 follow 모드를 켬 |
| `n` / `p` | 다음/이전 매칭 라인으로 이동 |
| `/` | 검색 입력창으로 포커스 이동 |
| `m` | 현재 커서 라인에 북마크 추가 또는 취소 |
| `w` | 로그 목록 wrap 모드 ON/OFF |
| `h` / `l` | wrap 모드가 꺼져 있을 때 로그 본문을 왼쪽/오른쪽으로 스크롤 |
| `1`-`9` | 해당 번호의 북마크 라인으로 이동 |
| `Space` | 현재 커서 라인의 Space pick 선택 토글 |
| `Shift+↑` / `Shift+↓` | 복사용 연속 구간 선택 확장 |
| `Shift+Home` / `Shift+End` | 복사용 연속 구간을 처음/끝까지 확장 |
| `c` | 선택된 로그를 클립보드에 복사 |
| `Esc` | 모드 스택의 최상위 모드를 취소하고 해당 모드 상태 clear |
| `F1` | 로그 목록 도움말 다이얼로그 표시 |
| `q` | 로그 목록 화면에서 종료 |
| `Ctrl+C` | 어디서나 종료 |

#### 7.2.1 모드 스택과 취소

로그 목록 화면은 선택 모드와 검색 모드를 stack으로 관리한다. 모드는 활성화된 순서대로 push하고, `Esc`는 최상위 모드 하나만 pop한다.

필수 규칙:

- 연속 구간 선택 또는 Space pick 선택이 시작되면 선택 모드를 push한다.
- 검색 문자열이 비어 있지 않은 상태로 적용되면 검색 모드를 push한다.
- 선택 모드를 pop하면 연속 구간 선택과 Space pick 선택 정보를 모두 clear한다.
- 검색 모드를 pop하면 적용된 검색 문자열과 검색 입력 edit buffer를 모두 clear한다.
- 선택 모드와 검색 모드가 모두 stack에 있으면 `Esc` 입력마다 하나씩 순차적으로 pop한다.
- 검색 입력창에서 편집 중인 상태의 `Esc`는 편집을 취소하고 기존 적용 검색은 유지한다.
- 도움말 팝업이 열린 상태의 `Esc`는 도움말만 닫고 모드 스택을 pop하지 않는다.

대용량 파일에서는 현재 화면 주변 window만 읽어야 한다. 전체 파일을 매 이동마다 다시 스캔하면 안 된다.

### 7.3 화면 스크롤과 커서

커서는 현재 표시 목록에서 선택된 로그 줄을 가리킨다. 화면 스크롤은 커서가 보이는 범위 안에 있도록 조정한다.

필수 규칙:

- `j` 또는 `↓`는 커서를 한 줄 아래로 이동한다.
- `k` 또는 `↑`는 커서를 한 줄 위로 이동한다.
- 커서가 화면 가장 하단 로그 행에 위치한 상태에서 아래로 이동하면 목록 화면이 한 줄 아래로 스크롤된다.
- 커서가 화면 가장 상단 로그 행에 위치한 상태에서 위로 이동하면 목록 화면이 한 줄 위로 스크롤된다.
- 커서가 화면 안쪽에서 위/아래로 이동할 수 있는 경우에는 스크롤 위치를 변경하지 않고 커서 위치만 이동한다.
- `PageUp` 또는 `Ctrl+B` 입력 시 커서가 화면 가장 상단 로그 행에 있으면 화면 단위로 위쪽으로 스크롤한다.
- `PageUp` 또는 `Ctrl+B` 입력 시 커서가 화면 가장 상단 로그 행에 있지 않으면 먼저 커서를 현재 화면 상단 로그로 이동하고 스크롤하지 않는다.
- `PageDown` 또는 `Ctrl+F` 입력 시 커서가 화면 가장 하단 로그 행에 있으면 화면 단위로 아래쪽으로 스크롤한다.
- `PageDown` 또는 `Ctrl+F` 입력 시 커서가 화면 가장 하단 로그 행에 있지 않으면 먼저 커서를 현재 화면 하단 로그로 이동하고 스크롤하지 않는다.
- `G`는 커서를 현재 출력된 마지막 로그 위치로 이동한다.
- `G` 처리 후 마지막 로그는 화면의 가장 마지막 로그 행에 위치해야 한다.
- `j`, `↓`, `Ctrl+F`, `PageDown`, `G`, `n` 등 커서 이동키 처리 결과 커서가 마지막 출력 로그에 위치하면 follow 모드가 자동으로 켜진다.
- 이동 대상이 SOT의 현재 범위를 벗어나면 가능한 가장 가까운 유효 위치로 clamp한다.

### 7.4 Follow 모드

follow 모드는 tail mode다. 새 로그가 SOT 파일에 기록되어 출력 가능해지면 커서가 항상 마지막 출력 로그에 위치하고, 화면도 마지막 출력 로그가 보이도록 스크롤된다.

필수 규칙:

- `stdio` 입력에서 새 로그가 SOT 파일에 append되면 follow 모드일 때 화면은 자동으로 마지막 로그를 따라간다.
- 파일 입력에서도 대상 파일이 증가하는 것을 감지할 수 있으면 follow 모드일 때 같은 방식으로 마지막 로그를 따라간다.
- 마지막 출력 로그에 커서가 위치하면 follow 모드는 자동으로 켜진다.
- `G` 키를 누르면 커서는 마지막 출력 로그를 가리키고, 마지막 로그는 화면상의 가장 마지막 로그 행에 위치하며, follow 모드가 켜진다.
- `j`, `↓`, `Ctrl+F`, `PageDown`, `n` 등 커서 이동키에 의해 마지막 출력 로그로 이동한 경우에도 follow 모드가 켜진다.
- follow 모드가 켜져 있는 동안 커서는 항상 마지막 출력 로그에 위치한다.
- follow 모드가 켜져 있는 동안 마지막 출력 로그는 화면상의 가장 마지막 로그 행에 위치한다.
- `k`, `↑`, `Ctrl+B`, `PageUp`, `Home`, `p` 등 커서를 마지막 출력 로그가 아닌 위치로 이동시키는 키 입력은 follow 모드를 자동으로 끈다.
- follow 모드가 꺼져 있으면 새 로그가 도착해도 화면은 자동 스크롤되지 않는다.
- follow 모드가 꺼진 상태에서도 사용자의 수동 스크롤과 커서 이동은 정상 동작한다.

### 7.5 매칭 라인 이동

`n`과 `p`는 현재 검색 문자열 기준으로 SOT에서 다음/이전 매칭 라인으로 이동한다.

필수 규칙:

- 검색 문자열이 비어 있으면 `n`/`p`는 커서를 이동하지 않는다.
- `n`은 현재 커서보다 뒤에 있는 다음 매칭 라인으로 이동한다.
- `p`는 현재 커서보다 앞에 있는 이전 매칭 라인으로 이동한다.
- 목록 끝과 처음에서 반대편으로 순환하지 않는다.
- 검색 문자열이 있지만 더 이상 이동할 매칭 라인이 없으면 상태바에 `no match`를 짧게 표시한다.
- 실제로 커서가 이동한 경우 follow 모드는 꺼진다.
- 매칭 라인 탐색은 현재 메모리 window에만 한정하지 않고 SOT 기준으로 수행한다.

### 7.6 북마크

북마크는 화면 목록이 포커스를 가지고 있고 커서가 로그 라인에 위치한 경우에만 동작한다. 필터 입력 모드 등 텍스트 입력 포커스에서는 `m`과 숫자키를 북마크 키로 처리하지 않는다.

필수 규칙:

- `m`을 누르면 현재 커서 라인에 북마크를 추가한다.
- 북마크는 최대 9개까지 지원한다.
- 북마크 번호는 추가될 때마다 `1`부터 `9`까지 자동 증가한다.
- 9개가 모두 설정된 상태에서 북마크가 없는 라인에 `m`을 누르면 새 북마크를 추가하지 않는다.
- 현재 커서 라인에 이미 북마크가 있으면 `m`은 해당 북마크를 취소한다.
- 북마크 취소 후 다른 북마크의 번호는 재배열하지 않는다.
- 숫자키 `1`-`9`를 누르면 해당 번호의 북마크 라인으로 이동한다.
- 해당 번호의 북마크가 없으면 이동하지 않는다.
- 해당 북마크 라인이 현재 화면 목록에 표시되지 않는 경우에는 이동하지 않는다.
- 북마크 라인으로 실제 이동한 경우 follow 모드는 꺼진다.

### 7.7 로그 복사용 선택

로그 복사용 선택은 로그 목록 화면이 포커스를 가지고 있을 때만 동작한다. 필터 입력창과 검색 입력창에서는 `Space`, `c`, Shift 이동키를 선택 단축키로 처리하지 않는다.

용어:

- 연속 구간 선택: Shift 이동키로 설정한 앵커와 현재 커서 사이의 연속 로그 라인 집합.
- Space pick 선택: `Space`로 개별 토글한 비연속 로그 라인 집합.
- 선택 모드: 연속 구간 선택이 있거나 Space pick 선택이 1줄 이상 있는 상태.

필수 규칙:

- `Shift+↑` 또는 `Shift+↓`를 누르면 첫 Shift 이동 시점의 커서를 앵커로 삼고 현재 커서까지 연속 구간을 선택한다.
- `Shift+Home` 또는 `Shift+End`는 앵커부터 목록 처음/끝까지 연속 구간을 확장한다.
- 일반 `j`/`k`/`↑`/`↓` 이동은 Shift로 만든 활성 연속 구간 표시를 해제할 수 있지만, 그 구간에 포함된 로그 라인의 선택 상태는 Space pick 선택처럼 유지한다.
- `PageUp`, `PageDown`, `Ctrl+B`, `Ctrl+F`, `Home`, `End`, `n`, `p` 이동도 활성 연속 구간 표시만 해제할 수 있으며 이미 선택된 로그 라인은 유지한다.
- `Space`를 누르면 현재 커서 라인의 Space pick 선택을 토글한다.
- 연속 구간 선택이 활성화된 상태에서 `Space`를 누르면 해당 구간의 선택 상태를 유지한 뒤 현재 커서 라인의 선택을 토글하고 활성 연속 구간 표시를 해제한다.
- 필터 입력창이나 검색 입력창에 포커스가 있을 때 `Space`는 해당 입력창의 문자 입력으로 처리하고 선택 상태를 변경하지 않는다.
- `c`를 누르면 연속 구간 선택과 Space pick 선택의 합집합을 raw line number 오름차순으로 클립보드에 복사한다.
- 같은 로그 라인이 연속 구간과 Space pick 양쪽에 포함되어도 한 번만 복사한다.
- 연속 구간 선택과 Space pick 선택이 모두 없으면 `c`는 현재 커서 라인 하나만 복사한다.
- 필터 입력창이나 검색 입력창에 포커스가 있을 때 `c`는 해당 입력창의 문자 입력으로 처리하고 복사를 수행하지 않는다.
- 복사 성공 시 상태바 일시 메시지로 `N lines copied` 또는 `1 line copied`를 표시하고, 짧은 시간 후 자동으로 지운다.
- 복사할 로그가 없으면 상태바 일시 메시지로 `no lines`를 표시하고, 짧은 시간 후 자동으로 지운다.
- `Esc`는 로그 목록 화면에서 선택이 있으면 연속 구간 선택과 Space pick 선택을 먼저 해제한다.
- 로그 목록에서 `:`로 필터 입력창에 진입하거나 `/`로 검색 입력창에 진입하면 연속 구간 선택과 Space pick 선택을 모두 해제한다.
- 필터 입력창에서 `Enter`로 필터 적용에 성공하면 연속 구간 선택과 Space pick 선택을 모두 해제한다.
- 상태바는 연속 구간 선택과 Space pick 선택의 합집합을 하나의 `sel:<N>` 필드로 표시한다.

### 7.8 로그 목록 도움말

로그 목록 화면이 포커스를 가지고 있을 때 `F1`을 누르면 도움말 다이얼로그를 팝업창으로 표시한다.

필수 규칙:

- 도움말은 로그 목록 화면의 주요 키맵을 포함해야 한다.
- 도움말에는 이동, follow mode, 필터 입력, 검색 입력, 북마크, 선택/복사 키를 구역별로 표시한다.
- 도움말 다이얼로그가 표시되는 동안 로그 목록 커서, 스크롤, 필터, 검색, 선택 상태는 변경하지 않는다.
- 도움말 다이얼로그는 `Esc` 또는 `F1`로 닫을 수 있다.
- `?`는 도움말 키로 지원하지 않으며 로그 목록 화면에서는 도움말을 열거나 닫지 않는다.

## 8. 입력창 UX

### 8.1 필터 입력 진입

필터 입력 진입 키는 `:`만 사용한다.

명시 규칙:

- 로그 목록 화면에 커서가 있고 목록이 포커스를 가지고 있을 때 `:`를 누르면 포커스가 필터 입력창으로 이동한다.
- 필터 입력창으로 포커스가 이동하면 현재 적용된 필터 문자열을 편집 입력값으로 유지하고 커서는 문자열 끝에 위치한다.
- 로그 목록 화면에서 `Enter`는 필터 입력 모드로 진입하지 않는다.
- 필터 입력창에서 `Enter`는 현재 필터 입력값을 완료한다.
- `Enter` 입력 시 필터 문자열을 파싱하고, 성공하면 로그 목록 필터링에 적용한다.
- 필터 파싱에 실패하면 기존 적용 필터는 유지하고, 필터 입력창 포커스도 유지하며 오류를 표시한다.
- 필터 입력창에서 `Esc`는 현재 입력값을 유지한 채 이전 포커스로 돌아간다.
- 필터 입력창에서 `Down` 키를 누르면 다음 입력 영역인 검색 입력창으로 포커스가 이동한다. 이때 검색 입력창은 현재 적용된 검색 문자열을 편집 입력값으로 유지하고 커서는 문자열 끝에 위치한다.
- 필터 입력창에서 `F1`을 누르면 필터 도움말 다이얼로그를 팝업창으로 표시한다.
- 필터 입력창에서 `?`는 일반 문자 입력으로 처리하며 도움말을 열거나 닫지 않는다.
- 필터 도움말 다이얼로그를 닫으면 필터 입력창 포커스와 현재 입력값을 유지한다.

### 8.2 필터 동작

필터는 현재 SOT 기준으로 로그 목록에 남길 줄 집합을 결정한다. `stdio` 입력에서도 필터 대상은 메모리 버퍼가 아니라 SOT 파일에서 읽힌 로그 줄이다.

필수 규칙:

- 필터 문자열이 비어 있으면 전체 로그를 표시한다.
- 필터 문자열이 비어 있지 않으면 아래 필터 문법을 파싱해 통과하는 줄만 표시한다.
- 필터 적용 후 검색 문자열은 지우지 않는다.
- 적용 순서는 필터가 먼저이고, 검색 표시와 `n`/`p` 매칭 이동은 필터를 통과한 목록 안에서만 동작한다.
- 필터 파싱에 실패하면 기존 적용 필터는 유지하고, 필터 입력창에 오류를 표시한다.

#### 8.2.1 토큰화

- 필터 문자열은 공백 또는 탭으로 토큰을 나눈다.
- 큰따옴표(`"..."`)로 감싼 문자열은 공백을 포함하는 하나의 토큰으로 처리한다.
- 따옴표 파싱이 실패하면 필터 파싱 실패로 처리한다.
- 태그 값에 공백이 필요하면 태그 토큰 전체를 큰따옴표로 감싼다. 예: `"over_speed: false"`.
- 따옴표 없이 `tag:`로 끝나는 토큰 바로 다음 토큰이 `:`를 포함하지 않고 `|`도 아니면 두 토큰을 하나의 태그 토큰으로 병합한다. 예: `over_speed:` `false` -> `over_speed: false`.
- 값 안에 `:`가 들어가면 큰따옴표로 감싸 하나의 토큰으로 입력한다.

#### 8.2.2 토큰 종류

| 종류 | 형태 | 의미 |
|------|------|------|
| 단순 토큰 | `term`, `+term`, `-term` | 줄 전체에 대한 부분 문자열 포함/제외 |
| 태그 토큰 | `tag:value`, `+tag:value`, `-tag:value` | 로그 줄의 태그/KV 값 또는 예약 태그 값 비교 |
| OR 토큰 | `|` | 좌우 필터 식 중 하나라도 만족하면 통과 |

태그 토큰은 다음 형식을 따른다.

```text
^([+-]?)([A-Za-z0-9_/.\-]+):(.+)$
```

태그 토큰 규칙:

- 선행 `+` 또는 `-`가 없으면 `+`로 해석한다.
- `tag`와 첫 `:` 사이에는 공백이나 탭을 허용하지 않는다.
- `value`는 첫 `:` 뒤의 전체 문자열이며 비어 있으면 파싱 오류다.
- `+` 단독, `-` 단독, `tag:`처럼 불완전한 토큰은 파싱 오류다.

#### 8.2.3 결합 규칙

- 단순 포함 토큰(`term`, `+term`)은 줄 전체에 해당 문자열이 포함되어야 한다.
- 단순 제외 토큰(`-term`)은 줄 전체에 해당 문자열이 포함되면 줄을 제외한다.
- 여러 단순 토큰은 AND로 결합한다.
- 서로 다른 태그 키 조건은 AND로 결합한다.
- 같은 태그 키에 여러 `+` value가 있으면 해당 태그 축 안에서 OR로 결합한다.
- 같은 태그 키에 여러 `-` value가 있으면 어느 하나라도 일치할 때 제외한다.
- 같은 `(tag, value)` 쌍에 대해 `+`와 `-`가 모두 나오면 필터 문자열에서 먼저 나온 토큰만 유효하고 뒤쪽 중복 토큰은 무시한다.
- 같은 태그 키에 포함 후보가 하나 이상 있으면 줄은 그 중 하나를 만족해야 한다.
- 제외 후보는 포함 후보 만족 여부와 무관하게 항상 우선 적용한다.

#### 8.2.4 최상위 OR

- 토큰화와 `tag:` 값 병합 이후, 토큰이 정확히 `|`인 위치에서 필터 식을 가지(branch)로 나눈다.
- 각 가지 안에서는 단순 토큰과 서로 다른 태그 키가 AND로 결합되고, 동일 태그 키의 다중 `+` 값은 OR로 결합된다.
- 줄은 한 가지라도 만족하면 전체 필터를 통과한다.
- 선행 `|`, 후행 `|`, 연속 `|`처럼 빈 가지가 생기면 파싱 오류다.
- `||`처럼 공백 없이 붙은 토큰은 OR 토큰이 아니며 파싱 오류로 처리한다.

#### 8.2.5 예약 태그 `level`

`level`은 예약 태그다. 로그 줄에 `level:` 문자열이 없어도 로그 타입별 규칙으로 심각도를 추출해 비교한다.

로그 타입별 level 추출:

| 로그 타입 | 추출 규칙 |
|-----------|-----------|
| `plain` | 구조적 level 추출 없음. `level:` 필터는 추출 실패로 취급 |
| `adb` | Android logcat의 `V/D/I/W/E/F` 문자를 `VERBOSE/DEBUG/INFO/WARN/ERROR/FATAL`로 정규화 |
| `kernel` | `[...] LEVEL:` 형태의 bracket level을 우선 추출. 추출되지 않으면 level 없음 |

`level` 비교 규칙:

- `level:ERROR`처럼 입력한 value는 추출된 정규화 level과 비교한다.
- level 추출에 실패한 줄은 `+level:value` 포함 조건이 하나라도 있으면 통과하지 않는다.
- `-level:value` 제외 조건만 있는 경우, level 추출 실패 줄은 제외 value와 일치하지 않은 것으로 본다.

#### 8.2.6 일반 태그 KV 동치

`level`이 아닌 태그는 로그 줄 안의 KV 표현과 비교한다.

필수 규칙:

- 필터의 `tag:value`는 로그 줄의 `tag=value`, `tag:value`, `tag: value`와 동치로 비교한다.
- 로그 줄에서는 `tag = value`, `tag : value`처럼 키, 구분자, 값 주변의 공백과 탭을 허용한다.
- 키를 찾을 때 바로 앞 문자가 `[A-Za-z0-9_/.\-]`에 속하면 더 긴 식별자의 일부로 보고 매칭하지 않는다.
- 값은 구분자 뒤 공백/탭을 제거한 뒤 첫 콤마, 첫 유니코드 공백, 줄 끝 전까지로 추출한다.
- 비교 전 필터 value와 로그 value는 `TrimSpace`한다.
- 값 안에 콤마나 공백을 포함하는 KV 비교는 v1 범위 밖이다.

#### 8.2.7 대소문자

- 기본 필터 비교는 대소문자를 구분한다.
- `--ignore-case`가 켜져 있으면 단순 토큰, 태그 키 탐색, 태그 value 비교, `level` value 비교에 대소문자 무시를 적용한다.
- `--ignore-case`는 검색 문자열 표시와 `n`/`p` 검색 이동에는 적용하지 않는다.

#### 8.2.8 예시

| 필터 | 의미 |
|------|------|
| `timeout retry` | `timeout`과 `retry`를 모두 포함한 줄 |
| `timeout -retry` | `timeout`을 포함하고 `retry`는 포함하지 않은 줄 |
| `level:ERROR level:WARN` | level이 ERROR 또는 WARN인 줄 |
| `level:ERROR timeout` | level이 ERROR이고 `timeout`을 포함한 줄 |
| `service:prod err | service:prod warn` | `service=prod`이면서 `err`를 포함하거나, `service=prod`이면서 `warn`을 포함한 줄 |
| `"over_speed: false"` | 공백을 포함한 태그 value 비교 |

### 8.3 검색 입력창

검색 입력창은 로그 목록을 줄이지 않고, 현재 검색 문자열을 편집하는 입력 영역이다. 현재 검색 문자열은 매칭 라인 이동(`n`/`p`)과 검색 표시의 기준이 된다.

명시 규칙:

- 로그 목록 화면에 커서가 있고 목록이 포커스를 가지고 있을 때 `/`를 누르면 포커스가 검색 입력창으로 이동한다.
- 검색 입력창으로 포커스가 이동하면 현재 적용된 검색 문자열을 편집 입력값으로 유지하고 커서는 문자열 끝에 위치한다.
- 검색 입력창에서 `Enter`는 현재 검색 입력값을 완료하고 현재 검색 문자열로 적용한다.
- 검색 입력값이 빈 문자열이면 현재 검색 문자열을 비운다.
- 검색 입력창에서 `Esc`는 이전 검색 입력을 유지하고 이전 포커스로 이동한다.
- `Esc`로 나갈 때 현재 편집 중인 검색 입력값은 적용하지 않는다.
- 검색 입력창에서 `Up` 키를 누르면 필터 입력창으로 포커스가 이동한다. 이때 필터 입력창은 현재 적용된 필터 문자열을 편집 입력값으로 유지하고 커서는 문자열 끝에 위치한다.
- 검색 입력창에서 `Down` 키를 누르면 로그 목록창으로 포커스가 이동한다.
- 검색 입력창에서 `F1`을 누르면 검색 도움말 다이얼로그를 팝업창으로 표시한다.
- 검색 입력창에서 `?`는 일반 문자 입력으로 처리하며 도움말을 열거나 닫지 않는다.
- 검색 도움말 다이얼로그를 닫으면 검색 입력창 포커스와 현재 편집 중인 검색 입력값을 유지한다.

## 9. 로그 파이프라인

### 9.1 SOT 저장 파이프라인

모든 UI 렌더링과 탐색의 기준은 SOT 파일이다.

필수 규칙:

- 파일 입력에서는 사용자가 지정한 원본 파일이 SOT다.
- `stdio` 입력에서는 별도 writer coroutine 또는 worker가 표준 입력을 SOT 파일에 먼저 저장한다.
- `stdio` writer는 UI update/render와 독립적으로 동작해야 하며, 입력 저장 때문에 UI가 blocking되지 않아야 한다.
- `stdio` 입력에서 UI는 표준 입력 스트림을 직접 표시 소스로 사용하지 않고, 저장된 SOT 파일을 읽는 source를 사용한다.
- SOT 파일에 기록되지 않은 로그는 화면에 표시하지 않는다.
- SOT 파일을 기준으로 삼아 프로세스 내부 버퍼 evict, UI 지연, 순간적인 backpressure 상황에서도 로그 분실 가능성을 최대한 낮춘다.

### 9.2 Raw Page 로딩

SOT 파일은 raw page 단위로 읽는다. raw page 크기는 64 KiB다.

필수 규칙:

- 파일 읽기는 64 KiB raw page 단위로 수행한다.
- 슬라이딩 탐색을 위해 현재 커서 위치가 속한 page와 전후 page를 함께 유지한다.
- 메모리에 로드되는 raw page는 최대 3개다.
- 3개 raw page는 커서 주변의 연속 page window다.
- 다음 page가 로드되어 3개를 초과하면 가장 오래된 반대편 page를 drop해 항상 최대 3 raw pages를 유지한다.
- page window를 앞으로 이동할 때는 `0 -> 1 -> 2 -> ... -> 마지막` 순서로 sequential하게 로딩한다.
- page window를 뒤로 이동할 때는 `마지막 -> ... -> 2 -> 1 -> 0` 역순으로 sequential하게 로딩한다.
- page `0`은 파일의 첫 64 KiB 영역이다.

### 9.3 Page 메타데이터

각 raw page는 로드할 때 라인 메타데이터를 계산한다.

필수 메타데이터:

| 값 | 설명 |
|----|------|
| page index | SOT 파일에서 64 KiB 단위로 나눈 0-based page 번호 |
| byte range | page가 담당하는 SOT 파일 byte 범위 |
| first raw line number | 해당 page의 첫 번째 로그 라인 번호 |
| raw line count | 해당 page에 속한 로그 라인 수 |

라인 번호 규칙:

- 첫 번째 page의 첫 번째 로그 라인은 raw line number `1`이다.
- raw line number `0`은 로그가 없음을 의미하는 sentinel 값이다.
- page에 로그 라인이 하나도 없으면 `first raw line number = 0`, `raw line count = 0`으로 둔다.
- 로그 한 줄이 page 경계에 걸쳐 존재하면 그 로그는 먼저 시작된 page에 속한다.
- page 경계에 걸친 로그 때문에 다음 page의 시작 raw line number는 해당 로그 다음 라인 번호부터 시작한다.
- raw page가 메모리에서 drop되어도 해당 page의 `first raw line number`와 `raw line count` 메타데이터는 유지한다.

### 9.4 Random Access

raw line number를 알면 해당 라인을 포함하는 page window를 로드할 수 있어야 한다.

필수 규칙:

- 유지 중인 page 메타데이터를 사용해 raw line number가 속한 page index를 찾는다.
- raw line number가 속한 page를 중심으로 이전/current/다음 최대 3개 raw pages를 로드한다.
- random access 로딩도 최종 메모리 상태는 최대 3 raw pages여야 한다.
- 아직 메타데이터가 없는 page로 이동해야 하면 필요한 방향으로 sequential scan을 진행해 page 메타데이터를 만든 뒤 로드한다.

### 9.5 필터링 파이프라인

필터링은 raw pages에서 line by line으로 수행한다.

용어:

- raw log: SOT 파일에서 읽은 원본 로그 라인.
- output log 또는 filter log: 필터 함수를 통과해 화면 출력 후보가 된 로그 라인.

필수 규칙:

- raw pages의 로그 라인을 순서대로 필터 함수에 전달한다.
- 필터 입력이 비어 있으면 필터 함수는 항상 `true`를 반환한다.
- 필터 입력이 있으면 §8.2 필터 문법으로 컴파일한 필터 함수를 사용한다.
- 필터 함수가 `true`를 반환한 raw log만 output log/filter log로 만든다.
- output log는 원본 raw line number를 보존해야 한다.

### 9.6 화면 출력 로그 캐싱

화면 표시를 위해 output log/filter log를 별도 cache에 보관한다.

필수 규칙:

- output log cache는 최대 10,000 lines를 유지한다.
- cache 항목은 필터를 통과한 로그 라인과 해당 raw line number를 포함한다.
- 10,000 lines를 초과하면 오래된 output log부터 삭제한다.
- 단, 현재 화면에 표시 중인 로그는 삭제하면 안 된다.
- 커서 위치를 기준으로 한 화면 이전의 시작 로그까지는 보호한다.
- 구체적으로 `protectedStart = max(0, cursorOutputIndex - viewportHeight)`로 계산하고, 이 index보다 오래된 output log만 drop 대상이 된다.
- drop 후에도 현재 커서 라인과 현재 화면에 표시되는 output log는 cache에 남아 있어야 한다.

## 10. 성능과 저장 요구사항

- `stdio` writer worker는 입력 순서를 보존해야 한다.
- SOT 파일 append 실패는 사용자에게 오류로 표시하고, 더 이상 해당 줄을 화면에 표시하지 않아야 한다.
- UI는 SOT 파일을 기반으로 bounded window/cache를 유지한다.
- 파일 입력은 파일 크기에 비례해 전체 메모리를 사용하지 않는다.
- raw page cache는 최대 3 * 64 KiB를 유지한다.
- output log cache는 최대 10,000 lines를 유지한다.
- 자동 로그 타입 판별은 초기 샘플 범위만 읽어 수행한다.
- UTF-8 로그를 기본 대상으로 한다. 잘못된 UTF-8 바이트는 UI가 깨지지 않도록 대체 표시할 수 있다.

## 11. CLI 요구사항

최소 CLI:

```text
logsee [options] [input-file|-]
```

위치 인자:

| 인자 | 설명 |
|------|------|
| `[input-file|-]` | 로그 파일 지정. 지정하지 않거나 `-`이면 `STDIO`에서 로그를 읽어 들인다. |

필수 옵션:

| 옵션 | 설명 |
|------|------|
| `--config <path>` | `config.toml` 경로 지정 |
| `--log-type <auto|plain|adb|kernel>` | 입력 로그의 타입 지정 |
| `--out <path>` | `stdio` 입력 SOT 파일 경로 지정 |
| `--ignore-case` | 필터 토큰과 태그 비교에 대소문자 무시 적용 |

동작 예:

```sh
some-command 2>&1 | logsee
some-command 2>&1 | logsee --out session.log
logsee /var/log/syslog
logsee --log-type adb android.log
logsee --config ./config.toml kernel.log
```

## 12. 개발 방법

### 12.1 Epic 관리

- PRD 요구사항은 기능 단위로 나누어 `docs/epics/` 아래 Epic 문서로 관리한다.
- Scrum Master Agent(`sm-agent`)는 기능 범위나 개발 범위가 한 번의 안전한 구현 단위로 너무 크다고 판단되면 필요에 따라 Epic을 새로 생성하거나 기존 Epic을 여러 Epic으로 분해한다.
- Epic을 생성하거나 분해할 때는 각 Epic의 PRD 추적, 의존관계, 우선순위, Acceptance Criteria, Acceptance Tests를 명시하고 `PLAN.md`와 `PROGRESS.md`를 같은 변경에서 갱신한다.
- Epic 문서는 `docs/epics/README.md`의 포맷을 따른다.
- Epic 파일명은 `docs/epics/EPIC-<NNN>-<Summary>.md` 형식을 따른다. `NNN`은 `001`부터 시작하는 3자리 번호이고, `Summary`는 짧은 kebab-case 요약이다.
- 새 Epic을 작성할 때 초기 상태는 반드시 `BACKLOG`다.
- Epic 상태 값은 `BACKLOG`, `ON_DEV`, `ON_QA`, `HOLD`, `CANCEL`, `DONE` 중 하나다.
- 각 Epic은 의존 Epic, 우선순위, PRD 추적 섹션, Acceptance Criteria, Acceptance Tests를 포함해야 한다.
- 각 Epic에는 `Acceptance Criteria Tests Pass`가 명시되어야 한다.

### 12.2 계획과 진행 상태

- 각 Epic의 의존관계와 우선순위를 판단해 구현 대상을 선정한다.
- 구현 계획은 루트 `PLAN.md`에 작성하고, 우선순위와 실행 순서를 명시한다.
- 모든 Epic의 현재 상태는 루트 `PROGRESS.md`에 업데이트한다.
- Epic 상태 변경은 Scrum Master Agent(`sm-agent`)가 조율하며, 상태가 변경되면 해당 Epic 문서와 `PROGRESS.md`를 함께 갱신한다.
- 구현이 완료되면 Epic 문서에 완료 상태(`DONE`)와 완료 근거를 명시한다.

### 12.3 개발 방식

- 구현 대상 Epic은 TDD 방식으로 개발한다.
- 개발 시작 요청을 받은 Dev Agent(`dev-agent`)는 구현할 Epic을 직접 선택하지 않고 Scrum Master Agent(`sm-agent`)에게 진행할 Epic을 요청한다.
- `sm-agent`는 `PLAN.md`, `PROGRESS.md`, Epic 의존관계를 확인해 다음 개발 대상 Epic을 지정한다.
- `sm-agent`는 다음 개발 대상 지정 전에 대상 범위가 너무 크면 Epic을 분해하거나 새 Epic을 생성해 개발 가능한 크기로 재계획한다.
- `sm-agent`는 Epic 개발 순서를 조율한다.
- `sm-agent`는 다음 개발 대상 선정 시 활성 BUG 문서가 존재하면 신규 Epic보다 BUG를 우선 선정하고 `dev-agent`에게 수정 요청한다.
- `dev-agent`는 `sm-agent`가 지정한 Epic만 대상으로 개발을 시작한다.
- 개발 시작 전 `sm-agent`는 지정된 Epic을 해당 Epic 문서와 `PROGRESS.md`에서 `ON_DEV` 상태로 갱신하고 `dev-agent`에게 개발을 요청한다.
- `dev-agent`는 Epic 개발 요청을 받으면 개발 계획을 목록화해 해당 Epic 문서의 `Development Execution Plan`에 업데이트하고 개발을 시작한다.
- TDD 순서는 `Red -> Green -> Refactor`를 따른다.
- `Red`: Epic의 Acceptance Tests를 먼저 작성하고 실패를 확인한다.
- `Green`: 요구사항을 만족하는 최소 구현으로 테스트를 통과시킨다.
- `Refactor`: 테스트가 통과하는 상태를 유지하며 구조와 중복을 정리한다.
- Epic을 `ON_QA`로 이동하기 전 해당 Epic의 Acceptance Tests가 모두 통과해야 한다.
- `dev-agent`는 Epic 개발 완료 후 구현 요약, 테스트 결과, 변경 파일, 남은 위험, 요청 상태를 `sm-agent`와 `qa-agent`에게 전달한다.
- `sm-agent`는 `dev-agent`에게서 Epic 개발 상태를 전달받으면 해당 Epic 문서의 완료 기록과 상태를 변경하고, `PROGRESS.md`에도 업데이트한다.
- QA Agent(`qa-agent`)는 개발 완료되어 `ON_QA`가 된 Epic을 검수하고, 검수 결과를 Scrum Master Agent(`sm-agent`)와 Dev Agent(`dev-agent`)에게 전달한다.
- `qa-agent`는 Epic 검증 요청을 받으면 검증 계획을 목록화해 해당 Epic 문서의 `QA Verification Plan`에 업데이트하고 검수를 시작한다.
- `qa-agent`의 검수 결과에는 통과/실패, Acceptance Criteria 확인 결과, 실행한 테스트와 결과, 결함 또는 누락 요구사항, 최종 Epic 상태가 포함되어야 한다.
- `sm-agent`는 `qa-agent`에게서 Epic 검증 상태를 전달받으면 해당 Epic 문서의 상태를 업데이트하고, `PROGRESS.md`에도 업데이트한다.
- `sm-agent`는 `qa-agent`의 검증 결과를 해당 Epic 문서에 기록한다.
- 검증 실패 시 `qa-agent`는 실패 내용을 `dev-agent`에게 전달한다.
- 검증 실패 규모가 큰 경우 `qa-agent`는 `docs/bugs/` 아래 BUG 문서를 작성하고 Epic 검증 결과에서 해당 BUG 문서를 참조한다.
- Epic이 검증 완료되어 `DONE`이 되면 `sm-agent`는 다음 Epic을 선정해서 `dev-agent`에게 개발을 요청한다.
- Epic이 `DONE`이 된 뒤 다음 작업을 선정할 때도 `sm-agent`는 활성 BUG를 신규 Epic보다 우선한다.

### 12.4 개발 아키텍처

- 개발 아키텍처는 Clean Architecture를 따른다.
- 도메인 모델과 비즈니스 규칙은 외부 프레임워크, TUI, 파일 시스템, OS clipboard에 의존하지 않는다.
- 의존성 방향은 바깥 레이어에서 안쪽 레이어로만 향한다.
- 안쪽 레이어는 바깥 레이어의 구체 구현을 직접 import하지 않는다.
- 외부 I/O는 interface/port로 추상화하고, adapter에서 구현한다.
- TUI, CLI, 파일 시스템, clipboard, config loader는 outer adapter로 둔다.
- Epic 구현 시 Acceptance Tests는 가능한 한 안쪽 use case/domain부터 작성하고, adapter는 별도 테스트로 검증한다.

권장 레이어:

| Layer | 역할 |
|-------|------|
| Domain | 로그 라인, raw line number, page metadata, filter expression, bookmark 같은 순수 모델 |
| Use Case | SOT 읽기, raw page loading, filtering, output cache, navigation/follow, search, copy workflow |
| Interface Adapter | CLI option 변환, config parsing adapter, file source adapter, clipboard adapter, TUI view model 변환 |
| Framework/UI | 실제 TUI 프레임워크, OS 파일 I/O, terminal rendering, clipboard command |

### 12.5 Agent 운영

- 세션 시작 시 `PRD.md`, `PLAN.md`, `PROGRESS.md`를 먼저 읽어 현재 요구사항, 구현 계획, Epic 상태를 파악한다.
- `PLAN.md`에 현재 구현 대상 Epic이 명시되어 있으면 해당 `docs/epics/EPIC-<NNN>-<Summary>.md`도 함께 읽는다.
- Agent 구성은 Scrum Master Agent, Dev Agent, QA Agent로 나눈다.
- Scrum Master Agent(`sm-agent`)는 Agent Team 리더이며, 계획, 작업 배정, 의존관계 순서, 상태 전환, handoff 경로, plan/progress 일관성을 최종 조율한다.
- Dev Agent(`dev-agent`)와 QA Agent(`qa-agent`)는 `sm-agent`의 배정과 조율을 따르고, 완료/검증/블로커/범위 변경 정보를 `sm-agent`에게 전달한다.
- Agent Team 리더 역할은 역할 경계를 우회하지 않는다. `sm-agent`는 명시적으로 해당 역할을 배정받지 않는 한 production code 구현이나 QA 검증을 직접 수행하지 않는다.
- 각 Agent는 반복되는 비슷한 작업이나 절차가 확인되면 해당 절차를 SKILL로 정의해 생성하고, 이후 같은 유형의 작업에서는 해당 SKILL을 우선 사용한다.
- SKILL은 트리거 조건, 최소한의 절차, 필요한 경우 scripts/references/assets 같은 재사용 자원을 포함하며, 반복 절차를 장황한 대화 기억이나 임시 문서에 의존하지 않도록 한다.
- SKILL 생성이나 갱신은 제품 요구사항, Epic 상태, QA 결과를 암묵적으로 바꾸지 않는다. 프로젝트 프로세스가 바뀌면 관련 문서도 함께 갱신한다.
- Scrum Master Agent(`sm-agent`)는 PRD, Epic 문서, `PLAN.md`, `PROGRESS.md`를 관리하고 의존관계와 우선순위, Epic 개발 순서를 조율한다.
- Scrum Master Agent(`sm-agent`)는 기능 범위나 개발 범위가 너무 큰 경우 Epic을 추가 생성하거나 분해하고, 그 결과를 Epic 문서, `PLAN.md`, `PROGRESS.md`에 일관되게 반영한다.
- Scrum Master Agent(`sm-agent`)는 다음 개발 작업 선정 시 활성 BUG가 있으면 BUG를 우선 선정해 Dev Agent(`dev-agent`)에게 수정 요청한다.
- Dev Agent(`dev-agent`)는 개발 시작 요청을 받으면 Scrum Master Agent(`sm-agent`)에게 진행할 Epic을 요청하고, 배정받은 `ON_DEV` Epic에 개발 계획을 기록한 뒤 TDD 방식으로 구현한다.
- Dev Agent(`dev-agent`)는 Epic 개발 완료 시 진행사항을 Scrum Master Agent(`sm-agent`)와 QA Agent(`qa-agent`)에게 전달한다.
- Scrum Master Agent(`sm-agent`)는 개발 완료 전달을 받으면 해당 Epic 문서와 `PROGRESS.md`의 상태를 갱신한다.
- QA Agent(`qa-agent`)는 개발 완료된 `ON_QA` Epic의 Acceptance Criteria와 Acceptance Tests를 검증하고, 검수 결과를 Scrum Master Agent(`sm-agent`)와 Dev Agent(`dev-agent`)에게 전달한다.
- QA Agent(`qa-agent`)는 Epic 검증 요청을 받으면 검증 계획을 Epic 문서에 기록한 뒤 검수를 시작한다.
- Scrum Master Agent(`sm-agent`)는 QA 검수 결과를 받으면 해당 Epic 문서와 `PROGRESS.md`의 상태를 갱신하고, 검증 완료된 Epic이 `DONE`이면 다음 Epic을 선정해 Dev Agent(`dev-agent`)에게 개발을 요청한다.
- Agent 역할과 세션 시작 규칙은 루트 `AGENTS.md`를 따른다.

### 12.6 프로젝트 구조 생성

첫 번째 구현 Epic은 프로젝트 구조 생성이다. 이 Epic은 기능 구현 전에 Go 모듈, 테스트 실행 기반, Clean Architecture 패키지 경계를 먼저 만든다.

필수 구조:

| 경로 | 용도 |
|------|------|
| `cmd/logsee/` | 실제 `logsee` 실행 파일 entrypoint |
| `internal/domain/` | 외부 I/O에 의존하지 않는 순수 모델과 규칙 |
| `internal/usecase/` | 입력, SOT, page loading, filtering, navigation 같은 기능 흐름 |
| `internal/port/` | use case가 사용하는 외부 I/O port/interface |
| `internal/adapter/cli/` | CLI option과 실행 구성 변환 |
| `internal/adapter/config/` | `config.toml` 로딩과 검증 adapter |
| `internal/adapter/filesystem/` | 파일/SOT 읽기 쓰기 adapter |
| `internal/adapter/tui/` | terminal UI adapter |
| `internal/adapter/clipboard/` | OS clipboard adapter |
| `configs/` | 기본/예시 설정 파일 |
| `testdata/` | 테스트용 로그 fixture |
| `docs/bugs/` | QA에서 발견한 큰 규모 결함 추적 문서 |

필수 규칙:

- `go.mod`를 생성해 `go test ./...`를 실행할 수 있어야 한다.
- `cmd/logsee`는 bootstrap과 adapter 연결만 담당하고 domain/use case 규칙을 직접 구현하지 않는다.
- `internal/domain`은 Go 표준 라이브러리의 순수 기능 외에 파일 시스템, 터미널, clipboard, TUI 라이브러리에 의존하지 않는다.
- `internal/usecase`는 `internal/port` 또는 use case 내부 interface에만 의존하고 adapter 구현을 import하지 않는다.
- adapter 패키지는 domain/use case 모델을 외부 라이브러리나 OS API에 맞게 변환한다.
- 프로젝트 구조 Epic의 Acceptance Tests는 패키지 import 방향과 전체 테스트 실행 가능성을 검증해야 한다.
